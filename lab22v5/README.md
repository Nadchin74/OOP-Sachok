# Лабораторна робота №22: Принцип підстановки Лісков (LSP)

## Опис початкової ієрархії та аналіз порушення
У початковій версії клас `BlackAndWhitePrinter` успадковувався від `Printer`. 
Базовий клас мав метод `PrintColorImage()`. 

**Порушення LSP полягало в тому, що:**
- Об'єкт підкласу не міг замінити об'єкт базового класу без збою в програмі.
- Контракт базового класу передбачав успішний кольоровий друк, але підклас викидав `NotSupportedException`.
- Клієнтський код, який очікує на `Printer`, "ламався" при отриманні чорно-білої моделі.

## Опис рефакторингу
Обрано підхід **зміни ієрархії та розділення інтерфейсів (ISP + LSP)**:
1. Створено базовий інтерфейс `IPrinter` лише з тими методами, які є спільними для всіх (друк тексту).
2. Створено спеціалізований інтерфейс `IColorPrinter`, який розширює базовий.
3. Тепер `BetterBWPrinter` не містить методу для кольорового друку взагалі, що унеможливлює помилку на етапі компіляції, а не під час роботи програми.

## Висновки
Дотримання LSP дозволяє створювати передбачувані системи. Використання інтерфейсів замість жорсткого успадкування допомагає уникнути ситуацій, коли об'єкт "бреше" про свої можливості.
