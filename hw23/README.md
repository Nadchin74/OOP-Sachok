# Доповідь: Юніт-тестування: теорія та практика

## 1. Переваги та обмеження юніт-тестування порівняно з інтеграційним

**Юніт-тестування (Модульне тестування)** — це процес перевірки окремих, найменших частин коду (методів, класів) в ізоляції від решти системи. 

**Інтеграційне тестування**, натомість, перевіряє, як кілька модулів або вся система загалом працюють разом, включаючи роботу з базами даних, файловою системою та зовнішніми API.

### Переваги юніт-тестування:
* **Швидкість:** Юніт-тести виконуються за мілісекунди, оскільки не потребують підключення до БД чи мережі. Це дозволяє запускати їх постійно під час розробки.
* **Точна локалізація помилок:** Якщо юніт-тест падає, розробник точно знає, в якому саме методі чи рядку коду проблема. В інтеграційних тестах пошук причини може зайняти години.
* **Документація коду:** Добре написані тести слугують живою документацією. Вони показують, як саме має поводитися клас за різних умов.
* **Спрощення рефакторингу:** Наявність юніт-тестів гарантує, що зміна внутрішньої логіки коду не зламає існуючий функціонал.

### Обмеження юніт-тестування:
* **Сліпота до системних помилок:** Юніт-тести не гарантують, що система працюватиме в зборі. Два класи можуть успішно проходити свої юніт-тести, але конфліктувати при спробі обміну даними (саме це і вирішують інтеграційні тести).
* **Витрати часу на мокування:** Для ізоляції класу часто доводиться писати багато "заглушок" (mocks/stubs), що може ускладнити підтримку тестів при зміні архітектури.

## 2. Практичний приклад: Клас та Тести

Нижче наведено простий клас `BankAccount` (Банківський рахунок) мовою C# з двома методами: поповнення та зняття коштів. 

### Клас `BankAccount`
```csharp
using System;

public class BankAccount
{
    public decimal Balance { get; private set; }

    public BankAccount(decimal initialBalance)
    {
        Balance = initialBalance;
    }

    // Метод 1: Поповнення
    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Сума поповнення має бути більшою за нуль.");
            
        Balance += amount;
    }

    // Метод 2: Зняття
    public void Withdraw(decimal amount)
    {
        if (amount > Balance)
            throw new InvalidOperationException("Недостатньо коштів на рахунку.");
            
        if (amount <= 0)
            throw new ArgumentException("Сума зняття має бути більшою за нуль.");

        Balance -= amount;
    }
}
```
### Юніт-тести (з використанням xUnit)
Для кожного методу написано два тести: один для успішного сценарію (Happy Path) і один для граничного випадку (Edge Case).

```csharp
using System;
using Xunit;

public class BankAccountTests
{
    // --- Тести для методу Deposit ---

    [Fact]
    public void Deposit_ValidAmount_IncreasesBalance()
    {
        // Arrange (Підготовка)
        var account = new BankAccount(100m);

        // Act (Дія)
        account.Deposit(50m);

        // Assert (Перевірка)
        Assert.Equal(150m, account.Balance);
    }

    [Fact]
    public void Deposit_NegativeAmount_ThrowsArgumentException()
    {
        // Arrange
        var account = new BankAccount(100m);

        // Act & Assert (Для виключень дія і перевірка часто поєднуються)
        Assert.Throws<ArgumentException>(() => account.Deposit(-10m));
    }

    // --- Тести для методу Withdraw ---

    [Fact]
    public void Withdraw_ValidAmount_DecreasesBalance()
    {
        // Arrange
        var account = new BankAccount(100m);

        // Act
        account.Withdraw(40m);

        // Assert
        Assert.Equal(60m, account.Balance);
    }

    [Fact]
    public void Withdraw_AmountGreaterThanBalance_ThrowsInvalidOperationException()
    {
        // Arrange
        var account = new BankAccount(50m);

        // Act & Assert
        Assert.Throws<InvalidOperationException>(() => account.Withdraw(100m));
    }
}
```
### 3. Використання Mock-об’єктів (Заглушок)

**Mock-об'єкти** — це імітація реальних залежностей класу. Вони дозволяють тестувати об'єкт в ізоляції, контролюючи поведінку його "сусідів".

#### Коли доцільно використовувати mock-об'єкти:
* **Зовнішні сервіси та API:** Якщо клас робить HTTP-запит до платіжної системи, у юніт-тесті ви не повинні реально звертатися до інтернету. Ви мокуєте `IHttpClient`, змушуючи його повертати фіксовану відповідь.
* **Робота з базою даних:** Для юніт-тестів не піднімають реальну БД. Інтерфейси репозиторіїв (наприклад, `IUserRepository`) мокуються, щоб одразу повертати тестові дані з пам'яті.
* **Недетермінована поведінка:** Робота з поточним часом (`DateTime.Now`), генераторами випадкових чисел або файловою системою. Наприклад, щоб протестувати логіку нарахування пені після закінчення терміну, ви мокуєте постачальника часу (`IDateTimeProvider`).
* **Важкі або повільні операції:** Відправка email, генерація великих PDF-звітів тощо.

#### Коли можна обійтися без mock-об'єктів:
* **Прості об'єкти значень (Value Objects) та DTO:** Немає сенсу мокувати класи-моделі (наприклад, `User` або `Transaction`), які лише зберігають стан і не мають складної логіки чи залежностей. Їх краще просто створювати через оператор `new`.
* **Чисті функції та утиліти:** Якщо клас залежить від іншого класу, який є простим калькулятором (приймає значення і повертає результат, не змінюючи стан системи), часто простіше використати реальний об'єкт.
* **Небезпека "Over-mocking":** Якщо тест складається переважно з налаштування моків, а не з перевірки логіки самого класу, це означає, що тест перевіряє *реалізацію*, а не *результат*. У таких випадках тест стає крихким і ламається при будь-якому внутрішньому рефакторингу.
